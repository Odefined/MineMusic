# Projection Maintenance Uses pg-boss Native Retry With includeMetadata Observation

Accepted. Projection maintenance rebuilds dirty projection targets as pg-boss jobs that rely on pg-boss's native retry (`retryLimit`/`retryDelay`/`retryBackoff` submitted with each job) rather than an in-handler retry loop. The pg-boss worker runs with `includeMetadata: true`, so the work handler observes `retryCount`/`retryLimit` on each job and, on the final attempt (`retryCount >= retryLimit`), marks the target `failed` instead of rethrowing — preserving the `status='failed'` row as the terminal record. This differs from `library.import` (ADR-0029), which retries inside the handler. The split is deliberate: projection rebuild is idempotent (always re-reads the latest source-of-truth) and trusts pg-boss's backoff, so it gains nothing from per-attempt control; `library.import` advances a stateful provider-page cursor where write failures must not be retried and provider-read failures want explicit per-attempt backoff, so it keeps the loop in-handler. ADR-0029's premise that "the worker handler cannot observe `retryCount`" reflected the configuration at the time; pg-boss v12 exposes `retryCount`/`retryLimit`/`state` to the work handler via `includeMetadata` (`JobWithMetadata`), though it still emits no standalone job-failed event (no `onComplete`/`onFailure` hook in v12) — so retry-state observation happens inside the work handler, not via a separate failure callback. The `BackgroundWorkJob.retryCount`/`retryLimit` fields carry this pg-boss-specific signal today; a future backend swap would need an `isFinalAttempt` port abstraction to keep the handler backend-agnostic (tracked as follow-up).
