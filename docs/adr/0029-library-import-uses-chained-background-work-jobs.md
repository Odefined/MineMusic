# Library Import Uses Chained Background Work Jobs

Accepted. `library.import` drives its multi-page provider read as a chain of Background Work jobs — each job advances the import batch by one provider page and submits the next job when the provider still returns a `nextCursor` — replacing the previous caller-driven synchronous paging where the agent repeatedly called `continue`. `library.import.start` is fire-and-forget: it creates the batch and submits the first job, then returns immediately while the agent polls `library.import.status`; the batch record remains the source of truth for progress and completion. The chain paces itself with a fixed `runAfter` delay between pages to protect NCM/QQ provider rate limits. `start` is idempotent per running library: a partial unique index over `(owner_scope, provider_id, library_kind) WHERE status = 'running'` on the batch table enforces at most one running batch per library at the database boundary (excluding account, which is unresolved at insert time and would let concurrent starts through under Postgres NULL-distinct semantics), so a repeated `start` whose insert hits this constraint catches it and returns the existing running batch instead of spawning a concurrent self-driving chain that would double the provider request rate; a completed or failed batch releases the slot so a fresh import can start. Single-page provider read failures retry inside the handler with backoff rather than relying on pg-boss retries, because pg-boss emits no job-failed event and the worker handler cannot observe `retryCount`, so pg-boss retries would force a lazy read-time reconciliation in `status`; write failures are not retried. `library.import.continue` is removed: chained self-driving covers advancement and handler retry covers recovery, leaving a two-tool surface (`start`, `status`). This is viable because the platform library read is a stateless pure function (NCM cursor is an integer offset, QQ cursor is an integer page number, auth is resolved from provider config) and `processNextPage` is already a stateless, DB-driven advancement unit.

> Amended by ADR-0030: the premise "the worker handler cannot observe `retryCount`" was specific to the configuration at the time. pg-boss v12 exposes `retryCount`/`retryLimit`/`state` to the work handler via `work({ includeMetadata: true })` (`JobWithMetadata`), so retry state is now observable in-handler (see ADR-0030). `library.import` keeps its in-handler retry by deliberate choice — write failures must not be retried and provider-read failures want explicit per-attempt backoff control — not by the technical constraint stated above. Projection maintenance (ADR-0030) instead uses pg-boss native retry with includeMetadata observation, because its rebuild is idempotent and needs no per-attempt control.
